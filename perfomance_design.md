# Performance Design: Caching & Rate Limiting

## 1. Анализ производительности

### Архитектура системы
Система управления проектами (вариант 8) построена на:
- **PostgreSQL** - хранение пользователей, проектов, задач
- **MongoDB** - хранение комментариев с вложенными replies
- **JWT** - аутентификация пользователей

### Hot paths (часто выполняемые операции)

| Операция | Endpoint | Частота | Причина |
|----------|----------|---------|---------|
| Получение всех проектов | `GET /api/projects` | Очень высокая | Отображается при каждом входе пользователя |
| Поиск пользователя по маске | `GET /api/users/search` | Высокая | Назначение исполнителей задач |
| Получение комментариев задачи | `GET /api/tasks/{id}/comments` | Высокая | Открытие задачи с комментариями |
| Аутентификация | `POST /api/login` | Средняя | Каждый вход в систему |

### Медленные операции (с высокими задержками)

| Операция | Задержка | Причина |
|----------|----------|---------|
| Поиск по маске имени | ~70 мс (с индексом) | `LIKE '%...%'` даже с GIN индексом |
| Получение всех проектов | ~20-50 мс | Каждый запрос идет в PostgreSQL |
| GET /comments с replies | ~5-15 мс | Сетевой вызов MongoDB |
| JWT верификация | ~0.05-0.1 мс | Криптография (быстро) |

### Требования к производительности

| Тип операции | Требование (p95) |
|--------------|------------------|
| GET запросы (проекты, комментарии) | < 50 мс |
| Поиск пользователя по маске | < 80 мс |
| POST запросы (создание, обновление) | < 30 мс |
| Аутентификация | < 10 мс |


## 2. Стратегия кеширования

### Какие данные кешируются

| Данные | Причина кеширования | Частота изменений |
|--------|---------------------|-------------------|
| Список всех проектов | Часто читается, редко меняется | Низкая |
| Комментарии задачи | Часто читаются при открытии задачи | Средняя |

### Стратегия кеширования: Cache-Aside (Lazy Loading)
```
Запрос           -> Кеш?        -> БД
│         │
│ HIT     │ MISS
▼         ▼
Возврат │ Сохранить
из кеша │ в кеш  
```

Данные загружаются только при реальном запросе

### TTL 

| Endpoint | TTL | Обоснование |
|----------|-----|-------------|
| `GET /api/projects` | 300 секунд (5 минут) | Проекты создаются нечасто, можно хранить дольше |
| `GET /api/tasks/{id}/comments` | 60 секунд (1 минута) | Комментарии добавляются чаще, нужен более свежий кеш |

### Стратегия инвалидации

| Событие | Действие | Метод |
|---------|----------|-------|
| Создание нового проекта | Инвалидация кеша проектов | `Invalidate("projects:all")` |
| Добавление комментария | Инвалидация кеша комментариев задачи | `Invalidate("comments:task:X")` |
| Добавление ответа (reply) | Инвалидация кеша комментариев | `InvalidateByPrefix("comments:task:")` |

### Реализация кеша

```
// In-memory кеш (можно заменить на Redis при масштабировании)
class CacheManager {
    std::unordered_map<std::string, CacheEntry> cache_;
    std::mutex mutex_;
    
    std::optional<std::string> Get(const std::string& key);
    void Set(const std::string& key, std::string value, int ttl_seconds);
    void Invalidate(const std::string& key);
    void InvalidateByPrefix(const std::string& prefix);
};
```

## Стратегия Rate Limiting
POST /api/login	- Защита от брутфорс-атак  	 
POST /api/register - Защита от спам-регистраций  
GET /api/users/search - Тяжелый поиск по маске  
Реализовано: POST /api/login (как наиболее критичный)

### Алгоритм: Sliding Window
```
Временная шкала (окно 60 секунд):
|<------------------ 60 seconds ----------------->|
|                                                 |
[req][req][req][req]                    [new] → rejected
  ^                                       ^
  |                                       |
старые запросы                           лимит 5/мин
(удаляются из окна)                    (6-й запрос блокируется)
```

Точнее Fixed Window (нет всплесков на границах)

### Лимиты
| Endpoint | Лимит | Окно | Обоснование |
|----------|-------|------|-------------|
| `POST /api/login` | 5 попыток | 60 секунд | Человек не может ввести пароль 5+ раз за минуту |
| `POST /api/register` (планируется) | 3 попытки | 3600 секунд | Ограничение спам-регистраций с одного IP |
| `GET /api/users/search` (планируется) | 60 запросов | 60 секунд | Защита от тяжелых поисковых запросов |

### HTTP заголовки ответа
При превышении лимита возвращаются заголовки:
| Заголовок | Пример | Описание |
|-----------|--------|----------|
| `X-RateLimit-Limit` | `10` | Максимальное количество запросов в окне |
| `X-RateLimit-Remaining` | `0` | Оставшееся количество запросов |
| `X-RateLimit-Reset` | `45` | Секунд до сброса счетчика |
| `Retry-After` | `45` | Секунд до следующей попытки |
